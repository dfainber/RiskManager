1. Contexto e objetivo

O kit monitora diariamente o risco dos fundos multimercados da Galapagos (MACRO, QUANTITATIVO, EVOLUTION) para alimentar o Morning Call. O produto final é um HTML interativo gerado por generate_risk_report.py.

Métrica de sucesso: número de vezes por semana que o briefing leva a uma decisão do gestor. Se é lido e arquivado sem ação, o kit falhou.

Arquitetura: Python puro → PostgreSQL (GLPG-DB01) → HTML self-contained. Sem servidor, sem JavaScript framework — o HTML vive em disco local (data/morning-calls/) e é aberto no browser.

2. Arquitetura modular

O kit é 12 módulos Python especializados. Refactorado em 2026-04-22/23 (Fase 3) a partir de um monolito de 13k linhas. O orquestrador hoje registra deps, dispara os fetches em paralelo e concatena os HTML fragments produzidos pelos renderers — tudo o mais foi extraído.

Grafo de dependências

Bottom-up: runtime state na base, orquestrador no topo. Nenhum módulo importa do orquestrador.

Regras de import (top-down proibido)
  • Runtime state (risk_runtime) fica na base — quebra ciclos quando outros módulos importam DATA_STR como default arg
  • Pure data (risk_config) não depende de ninguém — seguro pra importar de qualquer lugar
  • Renderers importam de cima pra baixo — fund_renderers pode usar svg_renderers, mas não o contrário
  • Nenhum módulo importa do orquestrador — generate_risk_report.py é folha de saída, isola composição

Flow de execução

  1. main() em generate_risk_report.py: recebe DATA_STR via CLI, computa em risk_runtime.py
  2. Pré-warmup: fetch_all_latest_navs() popula _NAV_CACHE em bulk (economiza ~500-800ms)
  3. ThreadPoolExecutor(12) dispara 30+ fetch_* em paralelo (fetches em ~3.3s, total ~5.6s)
  4. Futures resolvem com fallback D-1 se dado do dia não existir
  5. Cálculos: compute_* de metrics.py processam os dataframes
  6. build_html() monta cada card via *_renderers → concatena em <section-wrap>
  7. Escreve HTML em data/morning-calls/YYYY-MM-DD_risk_monitor.html

Validação após mudanças

Refactors subsequentes seguem a skill safe-refactor (~/.claude/skills/safe-refactor/), que inclui:

  • smoke_test.py com numeric snapshot check (3580 valores pra baseline)
  • AST-level diff via ast.unparse() — prova zero semantic drift byte-a-byte
  • HTML back-to-back: regen com código antigo vs novo em <2min pra isolar refactor vs DB drift
  • Convenção documentada: tabelas PA/NAV/CDI da GLPG-DB01 re-escrevem continuamente; ~30-60 valores mudam a cada 5-10 min mesmo em datas passadas

4. Shell HTML: navegação, tabs, modal

Modelo mental de navegação

A UI tem 2 eixos: fundos × reports. Cada seção é uma tupla (fund_key, report_key, html) no section registry. O usuário navega em 3 modos:

  • Summary — dashboard cross-fund (em construção)
  • Por Fundo — seleciona fundo → vê todos os reports dele
  • Por Report — seleciona report → vê para todos os fundos numa caixa unificada

URL hash como single source of truth

Estado de navegação vive no hash: #summary, #fund=MACRO, #report=exposure. Permite bookmark e deep-link. A função applyState() é chamada em DOMContentLoaded e em hashchange.

Modal (Single-Name QUANT)

Overlay centrado com backdrop blur. Abre via botão em card (openSnModal()), fecha com ESC, click fora ou botão ×.

5. Report · Risk Monitor (VaR/Stress)

MACROQUANTEVOLUTION

Para cada fundo, duas métricas: VaR 95% 1 dia e Stress (cenário "Quant e Macro V").

Fórmulas

VaR % do NAV VaR% = −SUM(PARAMETRIC_VAR) / NAV × 100
Stress % do NAV Stress% = −SUM(stress_col) / NAV × 100
onde stress_col =   SPECIFIC_STRESS (MACRO, EVOLUTION)
  MACRO_STRESS    (QUANT)
Utilização (soft) util_soft = |métrica| / soft_limit × 100

Range bar (12M)

Para cada métrica, mostra posição dentro do min/max observado nos últimos 12 meses:

range_pct = (|val_hoje| − min_12M) / (max_12M − min_12M) × 100

Se range_pct ≥ 80% → zona de alerta (laranja + comentário automático em ALERT_COMMENTS).

6. Report · Stop Monitor (Risk Budget por PM)

MACRO

Orçamento de perda mensal por gestor: CI (Comitê), LF (Luiz Felipe), JD (Joca Dib), RJ (Rodrigo Jafet).

Regra de carrego

Base mensal = −63 bps. Stop do mês seguinte depende do PnL do mês atual:

Ganho (PnL > 0) carry = PnL × 50%
stop_next = −(63 + carry) // stop expande
Perda com stop "extended" (stop_atual > 63 bps e YTD positivo) consumed = loss_em_extra × 25% + loss_em_base × 50% + loss_além_do_stop × 100%
stop_next = −(63 − consumed)
Perda com stop "base" (stop ≤ 63 ou YTD negativo) consumed = loss_dentro_stop × 50% + loss_além_stop × 100%
stop_next = −(63 − consumed)

Caps (quando YTD negativo)

  • Semestral: stop_next ≥ −(128 − |YTD|)  ·  STOP_SEM = 128
  • Anual: stop_next ≥ −252  ·  STOP_ANO = 252

Gancho

Se consumed ≥ 63 (PnL do mês excedeu 1× o stop base), o mês seguinte abre com stop = 0. O mês seguinte ao gancho reseta para −63 sujeito aos caps.

Colunas do relatório

  • Margem (bps) = budget + PnL_MTD — o quanto ainda dá para perder antes do stop
  • Perf. MTD vs Stop = barra horizontal do PnL MTD contra o stop vigente
  • Histórico = YTD + chips SEM% e ANO% (consumo dos caps)
  • Status = 🟢 OK (margem > 30%), 🟡 ATENÇÃO (margem < 30%), 🔴 STOP (margem ≤ 0), ⚡ GANCHO
CI é exceção

O Comitê opera com stop em % do PL, não em bps: Soft = −1.5% (gatilho de redução, não liquidação), Hard = −2.33% (stop duro). Não tem regra de carrego.

7. Report · Exposição MACRO

MACRO

Decomposição hierárquica: fator de risco (RF) → produto; com toggle para visão POSIÇÕES (por RF) vs PM VaR (por gestor).

Fonte de exposição

SELECT "BOOK", "PRODUCT", "PRODUCT_CLASS", "PRIMITIVE_CLASS",
       SUM("DELTA")                 AS delta,
       SUM("DELTA" * "MOD_DURATION") AS delta_dur
FROM  "LOTE45"."LOTE_PRODUCT_EXPO"
WHERE "TRADING_DESK_SHARE_SOURCE" = 'Galapagos Evolution FIC FIM CP'
  AND "VAL_DATE"                  = DATE '{date}'
  AND "BOOK" ~* 'CI|JD|LF|QM|RJ'
  AND "BOOK" ~* 'Direcional|Relativo|Hedge|Volatilidade|SS'

Por que SHARE_SOURCE = Evolution? MACRO é alocado dentro do Evolution FIC. O filtro pega a parcela do Evolution que corresponde ao MACRO.

VaR (bps) alocado

O VaR por book (LOTE_BOOK_STRESS_RPM) vem em BRL. Distribui proporcionalmente ao delta entre produtos do mesmo RF:

VaR (bps) por produto — product-level prod_var(bps) = (delta_produto / delta_total_do_RF) × VaR_RF(bps)
VaR (bps) por PM pm_var(bps) = SUM(prod_var) sobre todos os produtos do PM
Unidade = bps (não %)

A coluna var_pct do código é calculada como var_brl × −10000 / NAV → resulta em basis points. A rotulação no HTML é "VaR (bps)". Soma dos PMs > VaR do fundo = benefício de diversificação (correlação < 1).

σ (bps, daily)

Fonte: q_models.STANDARD_DEVIATION_ASSETS, BOOK='MACRO'. Unidade: bps diários (i.e., σ × 100 em %). Exemplo: VALE3 σ=140 ⇒ 1.4% daily ⇒ ~22% annual (√252).

ΔExpo e ΔVaR

ΔExpo = pct_nav_D − pct_nav_D−1
ΔVaR = var_D − var_D−1

Cores: verde = aumentou (long), vermelho = diminuiu. Intensidade proporcional à magnitude.

DIA (PnL realizado)

Lookup (LIVRO, PRODUCT) em REPORT_ALPHA_ATRIBUTION.DIA × 10000. Agregado no nível PM somando todos os produtos do gestor.

8. Report · Single-Name (QUANT, decomposição WIN)

QUANT

Objetivo: mostrar exposição L/S por nome, decompondo o future WIN (Ibov) nos seus constituintes.

Passo 1 — posições diretas

SELECT "PRODUCT" AS ticker, SUM("DELTA") AS direct
FROM  "LOTE45"."LOTE_PRODUCT_EXPO"
WHERE "TRADING_DESK"              = 'Galapagos Quantitativo FIM'
  AND "TRADING_DESK_SHARE_SOURCE" = 'Galapagos Quantitativo FIM'
  AND "PRODUCT_CLASS" = 'Equity' AND "IS_STOCK" = TRUE
GROUP BY "PRODUCT"

Passo 2 — equity leg do WIN

O DELTA do future WINM26 vem em 2 filtros: PRIMITIVE_CLASS='BRL Rate Curve' (perna de juros) e 'Equity' (perna de ações, beta-adjusted). Soma-se apenas a equity leg:

SELECT SUM("DELTA") AS win_equity_delta
FROM  "LOTE45"."LOTE_PRODUCT_EXPO"
WHERE "TRADING_DESK_SHARE_SOURCE" = 'Galapagos Quantitativo FIM'
  AND "PRODUCT_CLASS"   = 'IBOVSPFuture'
  AND "PRIMITIVE_CLASS" = 'Equity'
Atenção — granularidade do DB

Cada BOOK reporta o WINM26 equity leg em ~83 rows (uma por constituinte IBOV já beta-decomposta pelo engine). A soma dessas 83 rows = exposição total beta-adjusted do book ao WIN. Validado: 42.63% de NAV para o QUANT em 16/04/2026.

Passo 3 — composição do IBOV

SELECT "INSTRUMENT" AS ticker, "VALUE" AS weight
FROM  public."EQUITIES_COMPOSITION"
WHERE "LIST_NAME" = 'IBOV'
  AND "DATE" = (SELECT MAX("DATE") FROM ... WHERE "DATE" ≤ data_ref)

Fórmula de decomposição

Decomposição do WIN por constituinte from_win(ticker) = win_equity_delta × weight_ibov(ticker)
Exposição net por nome net(ticker) = direct(ticker) + from_win(ticker)
Normalização net_pct_NAV = net × 100 / NAV

Sumário

  • Gross L = soma dos net positivos (em % NAV)
  • Gross S = soma dos net negativos
  • Net = Gross L + Gross S
  • Top 8 L / Top 8 S ordenados por |net|

9. Report · Distribuição 252d (Backward / Forward)

MACROEVOLUTION

Usa a tabela pré-computada q_models.PORTIFOLIO_DAILY_HISTORICAL_SIMULATION. Cada PORTIFOLIO_DATE produz uma série W dos últimos ~252 dias, onde W é o PnL hipotético (em bps) da carteira daquela data aplicada aos retornos de cada DATE_SYNTHETIC_POSITION.

Portfolios disponíveis

  • Fundo: MACRO, EVOLUTION
  • PM (MACRO): CI, LF, JD, RJ, QM
  • Fator (MACRO): RF-BZ, RF-DM, RF-EM, FX-BRL, FX-DM, FX-EM, RV-BZ, RV-DM, RV-EM, COMMODITIES, P-Metals
  • QUANT: Bracco, Quant_PA, SIST, SIST_COMMO, SIST_FX, SIST_RF, SIST_GLOBAL

Backward 252d

PORTIFOLIO_DATE = D−1 (carteira de ontem). Responde: "o movimento de hoje foi esperado ou extremo vs histórico?"

Percentil ordinal pct = |{i : W_i < DIA_today}| / N × 100
Número de desvios-padrão #σ = DIA_today / σ(W)

Colunas: Tipo · Nome · DIA · Percentil · #σ. Leitura: pct < 10 ou > 90 = cauda; |#σ| > 2 = evento raro.

Forward 252d

PORTIFOLIO_DATE = D (carteira de hoje). Responde: "dado a carteira que carrego agora, o que esperar em P&L diário?"

Estatísticas Min = min(W)  ·  Max = max(W)  ·  Média = mean(W)  ·  σ = std(W)
VaR 95 = percentile(W, 5) // cauda de perda
VaR +95 = percentile(W, 95) // cauda de ganho (simétrica)

Colunas: Tipo · Nome · Min · VaR 95 · Média · VaR +95 · Max · σ.

UI

Toggle Backward/Forward dentro do mesmo card, feito em JS (função setDistMode). CSV export pega só a view visível.

10. Regras universais · Sort + CSV

Sort automático em qualquer <th>

Função attachUniversalSort() roda em DOMContentLoaded, varre todas as tabelas e anexa handler de click nos headers. Para desligar numa tabela específica (ex: Risk Monitor com só 2 linhas):

<table class="metric-table" data-no-sort="1">

Tabelas com sort custom (Exposure usa sortTable(tbody, col, paired)) são detectadas e skipadas via onclick contendo sortTable(.

CSV export em cada card

Função injectCsvButtons() adiciona botão ⤓ CSV em todo .card-head e .modal-head. O CSV:

  • Inclui apenas tabelas visíveis (offsetParent !== null)
  • Respeita toggles (Backward/Forward, POSIÇÕES/PM VaR)
  • Nome: <card_title>_<date_picker_value>.csv
  • Encoding UTF-8 com BOM (\uFEFF) — abre bem no Excel-PT
  • Escape CSV padrão: aspas em valores com vírgula/ponto-e-vírgula/aspas
Regra do CLAUDE.md

Novas tabelas não precisam repetir nada — comportamento é injetado no DOMContentLoaded a partir das classes existentes.

EVOLUTION · regra D-1 obsoleta

A regra histórica "usar D-1 para EVOLUTION" foi removida em 2026-04-18. O delay D-1 é contábil (cota do FIC publicada em D+1 porque depende da cota do master), não de risco — o engine faz look-through nas posições do master e reporta same-day.

Validação empírica: 22 datas do último mês, VaR e NAV same-day em LOTE_FUND_STRESS_RPM e LOTE_TRADING_DESKS_NAV_SHARE. A regra ainda aplica a fundos fora do lote (estruturados com admin externo).

QUANT · nome canônico no DB

O fundo Galapagos Quantitativo FIM aparece como QUANT no banco (não SISTEMATICO, que era o nome antigo no Excel). Nos portfolios da simulação histórica ele é SIST e derivados (SIST_RF, SIST_FX, etc.).

12. Roadmap / pendências

Fase 4 — em andamento

  • Summary page — conteúdo (hoje placeholder). Heatmap fundos × métricas, top alerts, deltas D/D, orçamentos. É o Morning Call numa tela.
  • Mandatos ALBATROZ, IDKA_3Y, IDKA_10Y (aguardam definição de BVaR)

Fase 5 — analytics sugeridas (excluindo liquidez e crédito)

  1. Backtest de VaR — quantas vezes PnL real excedeu VaR 95% em 252 dias? Esperado ≈ 5%. É a fundação da calibração.
  2. Cross-fund / firm-level overlap — consolidar mesma posição aparecendo em vários fundos (QUANT dentro de Evolution, MACRO dentro de Evolution).
  3. Scenario library — shocks nomeados: "Fed +100bps", "BRL −5%", "S&P −3%", "Oil +20%".
  4. Drawdown trajectory — current DD from peak, tempo underwater, velocidade.
  5. Correlation breakdown — diversification benefit ao longo do tempo.
  6. Style drift — PM dentro do mandato (ex: RJ = Commodities; se aparece em RV-DM, alertar).
