2.1 Status consolidado

O que mostra: uma linha por fundo com retorno DIA/MTD/YTD/12M, VaR atual, utilização de VaR e de stop, e Δ VaR vs. D-1. No rodapé, 2 linhas "benchmark" (IBOV + CDI) para referência.

Cálculos

  • DIA/MTD/YTD/12M (alpha vs. CDI): soma de dia_bps/mtd_bps/ytd_bps/m12_bps da tabela q_models.REPORT_ALPHA_ATRIBUTION. Já net de CDI pela engine.
    Exceção: FRONTIER

    Usa TOTAL_IBVSP_{DAY,MONTH,YEAR} do mainboard Long Only (ER vs. IBOV, não vs. CDI).

  • VaR atual: última observação da série de risco por fundo. Convertido para % NAV, sempre positivo.
  • Util VaR = VaR atual / soft limit. Fundos com informative=True (ex.: FRONTIER) não têm util (não há limit).
  • Util Stop (só MACRO e ALBATROZ hoje): maior utilização de orçamento mensal entre PMs (MACRO) ou abs(MTD) / 150 bps (ALBATROZ).
  • Status 🟢/🟡/🔴: max(util VaR, util stop). 🟢 <70%, 🟡 70–100%, 🔴 ≥100%.
  • Δ VaR D-1 = |VaR hoje| − |VaR D-1|, em bps. "flat" se |Δ| < 0.5 bps.

Linhas benchmark (rodapé)

  • IBOV: fetch_ibov_returns() — 4 retornos compostos de public.EQUITIES_PRICES.CLOSE (INSTRUMENT='IBOV').
  • CDI: fetch_cdi_returns() — soma simples diária de public.ECO_INDEX (INSTRUMENT='CDI', FIELD='YIELD').

2.2 Risco Agregado (house-wide)

O que mostra: uma linha por fundo com NAV, VaR absoluto (% + R$), benchmark, BVaR relativo (% + R$). Linha total = soma simples (sem benefício de diversificação). Top-5 destacados: 🔺 maior abs R$ · 🔷 maior rel R$.

Cálculos

  • VaR abs (%): lido do series_map, mesma fonte do Status consolidado.
  • VaR abs (R$) = VaR% / 100 × NAV. NAV vem do _NAV_CACHE.
  • BVaR rel (%):
    • FRONTIER: compute_frontier_bvar_hs() — 3y HS vs. IBOV, com clip de corporate actions.
    • IDKA_3Y / IDKA_10Y: RELATIVE_VAR_PCT do engine (paramétrico, vs. IDKA index).
    • CDI-benchmarked (MACRO, QUANT, EVO, MACRO_Q, ALBATROZ): ≈ VaR abs, já que CDI tem vol diária ≈ 0.
  • Total row = soma simples. Conservador (assume correlação = 1 entre fundos).

Bruto vs. Líquido

Líquido (net of bench) liquido[factor, fund] = bruto[factor, fund] − bench_allocation[factor, fund]

Alocações de bench:

  • Real: IDKA 3Y bench = 3.0 × NAV, IDKA 10Y bench = 10.0 × NAV. Outros = 0.
  • IPCA Idx: IDKAs bench = 1.0 × NAV. Outros = 0.
  • Equity BR: Frontier bench = 1.0 × NAV (IBOV 100% long). Outros = 0.
  • Nominal / Commodities / FX / Equity DM/EM: todos bench = 0 (CDI tem zero duration).
Por que Líquido importa

IDKA 10Y tem +997M BRL·ano de exposição real bruta. O bench absorve 914M → líquido ~+83M. O risco ativo do fundo é 8× menor que o bruto sugere. No snapshot 2026-04-17, Real dropa de 1,250M → 161M (8× redução); IPCA Idx de 162M → 12M; Equity BR de 71M → 12M.

Convenção de sinal (fetch_rf_exposure_map)

# Para recover exposição de posição a partir do DELTA (sinal de hedge-side):
rate_prims = {"IPCA Coupon", "BRL Rate Curve"}
df["ano_eq_brl"] = df.apply(
    lambda r: -r["delta_brl"] if r["PRIMITIVE_CLASS"] in rate_prims else r["delta_brl"],
    axis=1,
)

-DELTA é equivalente a POSITION × MOD_DURATION para NTN-Bs e DAPs (verificado empiricamente). O engine guarda DELTA com sinal de hedge-side (negativo p/ long bond, representando o short-DI que hedgearia a posição). Negamos para recuperar a exposição real da posição.

2.4 Vol Regime

O que mostra: realized 21d vol (anualizada) vs. janela full HS, com pct rank e classificação (low / normal / elevated / stressed).

Cálculos

Formulas vol_recent_pct = std(W[-21:]) × √252 × 100
vol_full_pct = std(W) × √252 × 100
ratio = vol_recent / vol_full
pct_rank = percentil de std(W[t-21:t]) entre todas as janelas rolling de 21d

Fonte: q_models.PORTIFOLIO_DAILY_HISTORICAL_SIMULATION.W — série sintética de P&L simulada diária por portfolio (MACRO, SIST, EVOLUTION).

Regime:

  • low: pct_rank < 20
  • normal: 20 ≤ pct_rank < 70
  • elevated: 70 ≤ pct_rank < 90
  • stressed: pct_rank ≥ 90
Por que pct_rank, não z-score

Z-score do recente vs. distribuição rolling tem SE ~15% por overlap das janelas (N_eficaz ~36 em 756d). Pct_rank é não-paramétrico e imune ao overlap — sinal mais robusto.

2.5 Comments — Outliers do dia

O que mostra: por fundo, lista de produtos com contribuição anormal.

Condição de flag

flag = (|z| ≥ 2σ E |bps| ≥ 3) OU |bps| ≥ 10

A cláusula absoluta (|bps| ≥ 10) captura losses materiais em nomes historicamente voláteis (AXIA, etc.) onde σ é grande demais para o z-test disparar sozinho.

Cálculo

  • Fonte: q_models.REPORT_ALPHA_ATRIBUTION via fetch_pa_daily_per_product.
  • Para cada (FUNDO, LIVRO, PRODUCT): σ = std(dia_bps) dos últimos 90 dias.
  • z = dia_bps_today / σ.

4.2 ANO_EQ (1-year equivalent exposure)

Métrica canônica para exposição de taxas: quantos BRL de exposição ponderada por duration (ou equivalentemente: anos-equivalentes por 100% NAV) tem o fundo.

Fórmula canônica (calculadora existente) ANO_EQ = AMOUNT × DV01 × 10000 / AUM
= (AMOUNT × PU) × MOD_DURATION / AUM
= market_value × MOD_DURATION / AUM
= position_%NAV × MOD_DURATION

Nossa implementação em fetch_rf_exposure_map usa -DELTA para primitives IPCA Coupon / BRL Rate Curve, que é equivalente a POSITION × MOD_DURATION (verificado empiricamente).

Buckets de maturidade

_RF_BUCKETS = [
    ("0-6m", 0, 0.5), ("6-12m", 0.5, 1),
    ("1-2y", 1, 2), ("2-3y", 2, 3),
    ("3-4y", 3, 4), ("4-5y", 4, 5),
    ("5-6y", 5, 6), ("6-7y", 6, 7),
    ("7-8y", 7, 8), ("8-9y", 8, 9),
    ("9-10y", 9, 10), ("10y+", 10, 99),
]

Bucket = DAYS_TO_EXPIRATION / 365.25. Fallback: MOD_DURATION se days não disponível.

4.3 HS BVaR (Frontier vs. IBOV)

compute_frontier_bvar_hs(df_frontier, date_str, window_days=756):

  1. Pega pesos atuais do mainboard (% Cash por ticker, 17 ações hoje)
  2. Query 3y de CLOSE (756 business days + 120 buffer) para cada ticker + IBOV em public.EQUITIES_PRICES
  3. Pivoteia pra wide; drop linhas sem IBOV; forward-fill stocks em dias não-trading
  4. returns = wide.pct_change()
  5. Clip de corporate actions: returns[stock_cols].mask(|r| > 0.30, 0.0) — B3 tem circuit breaker individual em ~15%; qualquer coisa > 30% é split/bonificação (AMOB3 teve 5492% num dia, BBAS3 +50% — clipados)
  6. synthetic_fund_return = Σ weight_i × return_i
  7. ER = synthetic_fund_return − return_IBOV
  8. BVaR_95_1d = -quantile(ER, 0.05) × 100 (em %NAV)
Validação 2026-04-17

std(ER) = 0.51%/dia → parametric 1.645×std = 0.84% ≈ BVaR 0.85% (distribuição ~normal após clip). TE anualizada 8.1%/yr — plausível p/ LO 17-names.

4.4 BVaR paramétrico (IDKAs)

Direto de LOTE45.LOTE_PARAMETRIC_VAR_TABLE.RELATIVE_VAR_PCT. É a métrica do engine, vs. o índice IDKA correspondente (IDKA IPCA 3A / 10A).

4.5 Stop com carry forward (MACRO)

carry_step(budget_abs, pnl, ytd):

  • Se PnL ≥ 0: next_budget = STOP_BASE + pnl × 0.5 (mantém metade do ganho como cushion)
  • Se PnL < 0: consome cushion_extra × 0.25 + base × 0.50 + excesso × 1.0 (camada suave pro cushion, dura pro excesso)
  • Se YTD < 0: cap semestral STOP_SEM − |ytd| aplicado

Gancho (🔴) quando remaining ≤ 0. Constantes: STOP_BASE=63, STOP_SEM=128, STOP_ANO=252.

6 · Convenções

6.1 Idioma

  • Código, variáveis, JSON keys: inglês
  • UI/texto do Morning Call, comentários longos: português

6.2 Sinais

  • VaR/Stress: armazenado como PnL negativo (-R$), convertido para positivo em %NAV nos outputs
  • DELTA em LOTE_PRODUCT_EXPO para primitives de rate (IPCA Coupon, BRL Rate Curve): sinal hedge-side, negamos para obter exposição de posição
  • IPCA primitive: sinal direto (long bond = long IPCA face-value)

6.3 Unidades

  • VaR/Stress: % NAV ou bps de NAV (contexto dita)
  • ANO_EQ: anos-equivalentes (1.0 = 100% NAV em bond de 1y duration)
  • Retornos PA: % (2 decimais)
  • Exposição RF na tabela de Breakdown: BRL·ano para duration-based; BRL nocional para equity/commodity

6.4 NAV lag

NAV lag de ~1 business day vs. VaR/Expo. _latest_nav usa VAL_DATE ≤ date_str ORDER BY VAL_DATE DESC LIMIT 1. merge_asof(backward) em build_series para alinhar NAV aos dias de risco.

6.5 D-1 (sem shift contábil)

Não há "shift contábil D-1" — era um bug antigo (engine faz look-through para fundos nested). Todos os fundos usam D direto. Se dados de D faltarem, fallback explícito pra D-1 com banner ⚠ de stale.

7 · Como estender

7.1 Adicionar um novo fundo

  1. Mandato: criar data/mandatos/mandato-{SHORT}.json com soft/hard limits de VaR e Stress
  2. Dict: adicionar entrada em FUNDS (LEVEL=2/3 em RPM), RAW_FUNDS (product-level), ou IDKA_FUNDS (parametric BVaR)
  3. FUND_ORDER: adicionar o short em ordem desejada
  4. FUND_LABELS: adicionar display name
  5. _FUND_PA_KEY: adicionar pa_key (nome em REPORT_ALPHA_ATRIBUTION.FUNDO)
  6. Reports específicos: se precisar de algo além do default (Risk Monitor, PA, Análise), adicionar builder + registrar em sections.append((SHORT, "report-id", html))

7.2 Adicionar um novo report cross-fund

  1. Fetch: adicionar fetch_* + ex.submit no bloco paralelo
  2. Resolve: resolver o future com fallback
  3. Pass: passar como kwarg para build_html
  4. Build: criar build_* function que gera o HTML do card
  5. Register: injetar via f-string no summary_html
